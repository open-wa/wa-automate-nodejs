## Execution Order

The spec defines 9 phases. Due to the blast radius (15+ route files break simultaneously when schemas change), the execution order is adjusted to minimize broken-state windows:

| Step | Spec Phase | What | Breakage Risk |
|------|-----------|------|---------------|
| 1 | Phase 1 | Permission registry (`permissions.ts`, `defaults.ts`) | None — additive |
| 2 | Phase 3 | New `org-access.ts` helper | None — new file |
| 3 | Phase 2 | Schema changes (add `organizationId`, delete tables) | **Everything breaks** |
| 4 | Phase 4 | Route migration (all 15+ files) | Fixes breakage |
| 5 | Phase 9 | Delete dead middleware (`rbac.ts`, `role.ts`, `facility-access.ts`) | Cleanup |
| 6 | Phase 6 | Regenerate auth schema + DB migration | DB reset |
| 7 | Phase 7 | Seed script update | Seed works again |
| 8 | Phase 8 | Test framework migration (13 files, ~50 refs) | Tests compile |
| 9 | — | Build verification + grep for zero legacy refs | Gate |

> [!IMPORTANT]
> Steps 3 and 4 must happen as a single atomic sweep — schema deletion breaks compilation immediately, and all route files must be migrated before any build check can pass.
