# Migrate to Better Auth Multi-Org Architecture

## Background & Problem

Kurra has **three parallel ownership/access systems** that should be unified:

| Domain | Current Auth | Tables | Roles | Checks |
|---|---|---|---|---|
| **Facilities** | `facility.claimedByUserId` + `facility_staff` + `facility_invite` | 3 custom tables | owner / admin / operations | `requireFacilityPermission()` — 38 call sites across 8 files |
| **Squads** | `team_membership.role` + `team_invite` + `team_invite_draft` | 3 custom tables | captain / member | `checkIsCaptain()` — 20+ call sites in `teams.ts` |
| **Sponsors** | _Does not exist yet_ | — | — | — |

Better Auth's `organization` plugin is already installed with tables (`organization`, `member`, `invitation`) sitting empty. This plan unifies all three domains under BA orgs.

## Architecture: The Multi-Org-Type Model

### Core Insight: One Plugin, Many Org Types

Better Auth's organization plugin supports [`additionalFields`](https://www.better-auth.com/docs/plugins/organization#additional-fields-1) on the `organization` table. We add a **`type`** discriminator field:

```
organization.type = "facility" | "squad" | "sponsor"
```

Each org type gets its own set of roles and permissions, but they all share the same BA `member` table, `invitation` table, and `hasPermission()` API.

### Why Not Separate Systems?

- **Users are users.** A person can be a facility owner, a squad captain, AND a sponsor manager simultaneously. One BA `member` table tracks ALL their org memberships.
- **BA handles invites, roles, member CRUD.** We delete 6 custom tables (`facility_staff`, `facility_invite`, `team_membership`, `team_invite`, `team_invite_draft`, `team_follow`).
- **Forward-compatible.** Adding `"sponsor"` orgs later is just adding roles and permissions — zero schema changes.

### No `activeOrganizationId` — Explicit Org Per Request

> [!IMPORTANT]
> We will **NOT** use BA's `activeOrganizationId` session scoping. Reasons:
>
> 1. **Players browse without org context** — a user scrolling the feed doesn't need an active squad
> 2. **Cross-org views** — a facility owner viewing "all my facilities' bookings" spans multiple orgs
> 3. **Multi-tab ambiguity** — different tabs could need different org contexts
>
> Instead, our API routes will resolve the org from the **resource being acted on** (e.g., `facilityId` → look up `facility.organizationId` → check membership). The frontend will pass `organizationId` explicitly when needed.

### Phase 3: Org Permission Helper (API Layer)

#### [MODIFY] [facility-access.ts](file:///Users/Mohammed/projects/self/kurra-app/apps/api/src/lib/facility-access.ts) → Rename to `org-access.ts`

Replace the custom permission system with a thin wrapper around BA's [`auth.api.hasPermission()`](https://www.better-auth.com/docs/plugins/organization#access-control-usage):

```typescript
import type { Context } from "hono";
import { HTTPException } from "hono/http-exception";
import * as schema from "@kurra/db/schema";
import { eq } from "drizzle-orm";
import type { APIBindings } from "../middleware/type";

type OrgType = "facility" | "squad" | "sponsor";

/**
 * Check permissions on an org resolved from a resource.
 * Does NOT use activeOrganizationId — resolves org explicitly per request.
 *
 * @see https://www.better-auth.com/docs/plugins/organization#access-control-usage
 */
export async function requireOrgPermission(
  c: Context<APIBindings>,
  organizationId: string,
  permissions: Record<string, string[]>,
  forbiddenMessage: string
) {
  const auth = c.get("auth");

  // Temporarily set active org for the permission check
  // BA's hasPermission requires an active org in the session
  await auth.api.setActiveOrganization({
    body: { organizationId },
    headers: c.req.raw.headers,
  });

  const result = await auth.api.hasPermission({
    headers: c.req.raw.headers,
    body: { permissions },
  });

  if (!result?.success) {
    throw new HTTPException(403, { message: forbiddenMessage });
  }
}

/**
 * Resolve a facility's organizationId and check permissions.
 */
export async function requireFacilityPermission(
  c: Context<APIBindings>,
  facilityId: string,
  permissions: Record<string, string[]>,
  forbiddenMessage: string
) {
  const db = c.get("db");
  const facility = await db.query.facility.findFirst({
    where: eq(schema.facility.id, facilityId),
  });

  if (!facility?.organizationId) {
    throw new HTTPException(404, { message: "Facility not found" });
  }

  await requireOrgPermission(c, facility.organizationId, permissions, forbiddenMessage);
  return { facility };
}

/**
 * Resolve a squad's organizationId and check permissions.
 */
export async function requireSquadPermission(
  c: Context<APIBindings>,
  teamId: string,
  permissions: Record<string, string[]>,
  forbiddenMessage: string
) {
  const db = c.get("db");
  const team = await db.query.team.findFirst({
    where: eq(schema.team.id, teamId),
  });

  if (!team?.organizationId) {
    throw new HTTPException(404, { message: "Squad not found" });
  }

  await requireOrgPermission(c, team.organizationId, permissions, forbiddenMessage);
  return { team };
}

/**
 * Get all org IDs of a given type that the current user can access.
 * Replaces getAccessibleFacilityIdsForPermission.
 */
export async function getAccessibleOrgIds(
  c: Context<APIBindings>,
  orgType: OrgType
): Promise<string[]> {
  const auth = c.get("auth");
  const orgs = await auth.api.listOrganizations({
    headers: c.req.raw.headers,
  });

  if (!orgs) return [];

  return orgs
    .filter((org: any) => org.metadata?.type === orgType || org.type === orgType)
    .map((org: any) => org.id);
}
```


### Phase 5: Org Creation Flows

#### Facility Claiming

When a user claims a facility, we:
1. Call `auth.api.createOrganization({ body: { name: facility.name, slug: facility.id, type: "facility" } })`
2. Set `facility.organizationId = newOrg.id`
3. The creating user automatically becomes `owner`

#### Squad Creation

When a user creates a squad, we:
1. Call `auth.api.createOrganization({ body: { name: squad.name, slug: squad.inviteCode, type: "squad" } })`
2. Set `team.organizationId = newOrg.id`
3. Creator becomes `owner` (= captain)

#### Sponsor Creation (Future)

Same pattern — `type: "sponsor"`.

### Phase 6: Frontend Context

#### No Global "Mode" — Contextual Navigation

The frontend doesn't force a "facility mode" or "squad mode" globally. Instead:

- **Player context (default):** Browse feed, discover, make bookings → no org context needed
- **Facility management:** Navigate to `/facility/:id/...` → resolve org from facilityId
- **Squad management:** Navigate to `/teams/:id/...` → resolve org from teamId
- **Cross-org views:** `/settings` page shows all orgs → grouped by type

#### Client-Side Permission Checks

Use [`authClient.organization.hasPermission()`](https://www.better-auth.com/docs/plugins/organization#access-control-usage):

```typescript
// Before showing "Edit Campaign" button
const canManageCampaigns = await authClient.organization.hasPermission({
  permissions: { campaign: ["update"] },
  organizationId: facility.organizationId,
});
```

Or for static checks without server round-trip, use [`checkRolePermission()`](https://www.better-auth.com/docs/plugins/organization#access-control-usage):

```typescript
const canEdit = authClient.organization.checkRolePermission({
  permissions: { campaign: ["update"] },
  role: currentMember.role,
});
```

## Open Questions

> [!IMPORTANT]
> **1. Role naming strategy**: Should we use BA's default roles (`owner`/`admin`/`member`) and interpret them differently per org type, OR use explicitly named roles (`squad-captain`, `facility-admin`, etc.)? 
>
> **Option A: Shared roles** — simpler, fewer to register. `owner` means "captain" in squad context, "facility owner" in facility context. Permissions still differ per what the route checks.
>
> **Option B: Named roles** — explicit, self-documenting. `squad-captain` vs `facility-owner`. More roles to manage but zero ambiguity.
>
> **My recommendation: Option A** — the permissions (not the role names) define what you can DO. The org `type` field tells you WHAT KIND of org you're in. Named roles add complexity without much value since the same user won't be confused about their role within an org they're looking at.

> [!WARNING]
> **2. `setActiveOrganization` in the permission check**: The `requireOrgPermission` helper temporarily calls `setActiveOrganization` before `hasPermission`. This is because BA's `hasPermission` checks against the _active_ org. This means:
> - Every permission-checked request does one extra BA API call to set the active org
> - If two requests hit simultaneously for different orgs, there's a race condition on the session's `activeOrganizationId`
>
> **Alternative**: Query the `member` table directly (via Drizzle) to check role, then use `checkRolePermission()` which is synchronous and doesn't need an active org. This avoids the race condition entirely.
>
> **My recommendation**: Use the **direct Drizzle query + `checkRolePermission`** approach. It's one DB query, no race conditions, no coupling to BA's stateful session.

> [!NOTE]
> **3. BA Teams vs Kurra Squads**: BA's organization plugin has a "Teams" concept (sub-groups within an org). We are NOT using this. In Kurra, a "squad" IS an org (of type `squad`). BA Teams would be useful if we needed sub-groups WITHIN a squad, which we don't currently.
