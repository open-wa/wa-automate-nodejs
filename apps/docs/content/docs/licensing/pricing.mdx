# Pricing and licensing

This page is the decision surface for open-wa licensing. If you already know you
need a key and only want the runtime details, jump to
**[Licensed features](/docs/licensing/licensed-features)**.

<Callout type="note" title="What this page is for">
  Use this page when you are trying to decide whether the free/community surface
  is enough, whether you need a restricted license for gated flows, or whether
  you need insiders access for advanced or early-access capabilities.
</Callout>

## Actual prices or purchase paths

The public purchase path starts from the license key page:

Use that flow for current pricing, payment options, and the key issued for your
account or use case. The docs do not hard-code a price because the public
purchase flow is the source of truth for the current amount, available tiers,
and any support or insiders options shown at checkout.

If you are buying for a production workflow, confirm the exact restricted or
insiders feature first, then buy against the host-account session that will run
that workflow.

## What changes when you upgrade

<Steps>
  <Step title="Restricted licensing unlocks gated production use cases">
    This is the path when the community surface is not enough and you need a key
    for host-account-specific or gated runtime behavior.
  </Step>
  <Step title="Insiders licensing is for advanced or early-access capability">
    Choose this when the restricted tier is still too narrow and your workflows
    depend on higher-end or earlier-surface features.
  </Step>
  <Step title="The session context still matters">
    A license decision is not just about owning a key. The real runtime session,
    host account, and deployment context still determine whether the behavior you
    care about is available where you need it.
  </Step>
</Steps>

## Integration licensing

Integrations do not usually need a separate license just because they are
integrations. The license requirement follows the behavior they call.

For example, a webhook that only receives normal message events can stay on the
community surface. A Chatwoot, MCP, Node-RED, or custom plugin workflow that
calls a restricted method needs the tier that unlocks that restricted behavior.
If an integration depends on an insiders-only capability, treat the whole
workflow as insiders-scoped even if the integration package itself is available
to install.

<Callout type="tip" title="License the behavior, not the wrapper">
  Before buying for an integration, identify the exact API method, runtime flow,
  or host-account behavior the integration depends on. That is what determines
  the license tier.
</Callout>

## Team licensing

Team licensing should be tied to the shared production workflow, not only to the
developer who buys the key. If several people operate the same host-account
session or production deployment, decide who owns the billing path, where the key
is stored, and which runtime session the team is validating.

For organization or multi-seat needs, use the public license path first and treat
anything unclear as a buying conversation. Do not assume one key covers every
developer, account, environment, or customer deployment unless the purchase path
or support conversation confirms that scope.

## Renewal/cancellation

Renewal and cancellation are handled through the purchase or billing path used
to acquire the key. If the license is subscription-backed, manage renewal,
payment changes, or cancellation from that provider flow. If the purchase flow
does not expose the billing action you need, use the support or contact path tied
to the license purchase.

Operationally, plan renewals before a production dependency expires. Keep the
license key in deploy-time secret storage, rotate it when ownership changes, and
re-test the gated feature after any renewal, cancellation, or key replacement.
