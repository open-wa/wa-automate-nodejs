## HTTP auth/body conventions

Client routes are Easy API command routes mounted under `/api/`. If the Easy API process was started with `--api-key`, send that key on every command request with the `X-API-Key` header.

```bash
curl -X POST http://localhost:8080/api/messages/sendText \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-secure-key" \
  -d '{"to":"447123456789@c.us","content":"Hello from open-wa!"}'
```

Body-based routes use JSON objects. The object keys should match the `Name` column in the method parameter table. For example, `sendText` uses `to`, `content`, and optional `options`.

```json
{
  "to": "447123456789@c.us",
  "content": "Hello from open-wa!",
  "options": {
    "quotedMsg": "messageId"
  }
}
```

For `GET` routes with parameters, pass the listed values as query parameters unless a method page or generated schema says otherwise.

```bash
curl "http://localhost:8080/api/messages/getAll?chatId=447123456789@c.us" \
  -H "X-API-Key: your-secure-key"
```

Some parameter tables list key aliases, such as `text`, `body`, or `message` for `content`. Prefer the main parameter name for new code. Use key aliases only when you are keeping older integrations working or matching an existing request shape.

## Aliases explanation

Aliases exist so the current namespaced client surface can keep working with older route names and method names. The preferred route is listed as `HTTP route`, usually in the form `/api/{namespace}/{action}`. Alias routes are listed under `HTTP alias routes`, often in the older direct method form such as `/api/sendText`.

Method aliases work the same way. For example, a method may have a primary namespace method plus shorter or older names under `Aliases`. Deprecated aliases are still documented when the schema knows about them, but new code should avoid them.

Use this rule of thumb:

- New code should use the primary `HTTP route` and primary parameter names.
- Existing code can keep using alias routes while you migrate.
- Deprecated aliases should be treated as compatibility paths, not as examples to copy.
- If an alias and primary route disagree in style, trust the primary route for the current generated reference.

