# Docs ToC IA Red-Team Notes

Date: 2026-05-20
Scope: `apps/docs/content/docs/**/meta.json`

## Source Context

These notes capture the first-pass information architecture review of the Fumadocs page tree before reading GitHub issue #3337 directly.

The local issue #3337 persona source used in the first pass was `apps/docs/DOCS-QUALITY-CHECKLIST.md`, which says to use its persona table when historical issue comments are unavailable in the workspace.

## Persona Groups From Local Checklist

| Persona group | Navigation question |
| --- | --- |
| Junior Node dev | Can a new user run the first server, authenticate, send a first message, understand chat IDs, understand risky actions, and stay on the current API path? |
| AI/ML engineer | Can an AI integrator configure MCP or HTTP safely, see the auth boundary, and understand which AI examples are runnable versus patterns? |
| CRM/helpdesk operator | Can an operator choose webhook, Chatwoot, Node-RED, or S3 paths and understand the data flow before wiring a helpdesk? |
| Dictation plugin developer | Can a plugin author create, load, and test a dictation-style plugin without unsafe TypeScript patterns? |
| Moderation plugin developer | Can a moderation author understand post-detection action limits, failure behavior, secrets, and group filtering before shipping? |
| Copy red-team | Do changed pages start with user jobs, avoid internal nouns, put safety before capability, and include expected outputs where users copy commands? |

## Structural Findings

- All local MDX pages under `apps/docs/content/docs` were represented in their nearest parent `meta.json` `pages` array.
- No listed local pages were missing from disk.
- The docs tree is structurally complete; the main problem is journey friction, not broken navigation.
- Fumadocs `pages` arrays are explicit: when present, unlisted siblings do not appear unless included through `...` rest syntax.
- Fumadocs supports folder/page icons via `meta.json` or frontmatter `icon` strings when the loader maps them to React elements.
- `apps/docs/src/lib/source.ts` already maps icon strings through `lucide-static`.
- No docs `meta.json` or page frontmatter currently uses sidebar icons.
- `reference/client/meta.json` is generated by `packages/schema/scripts/gen-client-reference-docs.ts`; avoid hand-editing it.
- `reference/workspaces/**` is generated by `apps/docs/scripts/gen-workspace-readme-docs.js`; avoid hand-editing generated workspace metadata.

### Operations Split

Current root has both `operations` and `operations-and-troubleshooting`.

This is confusing because `operations` is titled `Security & Deployment` and contains one page, while `operations-and-troubleshooting` contains error handling, logout detection, and best practices.

Best IA direction:

- Prefer one `Operations` section containing security/deployment, best practices, error handling, and logout detection.
- If file moves are out of scope, at least rename `operations-and-troubleshooting` to `Troubleshooting` so the sibling distinction is honest.

## Icon Guidance

Use icons sparingly. Root folder icons can improve scanability; page-level icons in large sections will create noise.

Suggested root icons:

| Section | Icon |
| --- | --- |
| Getting Started | `Rocket` |
| Concepts | `BookOpen` |
| Guides | `Map` |
| Clients & Payloads | `Plug` |
| Plugins | `Puzzle` |
| API Reference | `FileCode` |
| Security & Deployment / Operations | `ShieldCheck` |
| Troubleshooting | `CircleAlert` |
| Licensing | `Scale` |

Avoid icons in generated reference sections. One justified page-level exception is `getting-started/v5-alpha`, where `FlaskConical` or `AlertTriangle` could signal preview/alpha status.
